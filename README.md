# Apple Bot (best.js)

`best.js`는 사과게임 보드를 읽어 자동으로 드래그를 수행하는 브라우저 콘솔용 스크립트입니다.

## 1) 사용법

### 빠른 실행
1. 사과게임 페이지를 연다.
2. 개발자 도구를 열고(Console 탭) `best.js` 전체 코드를 붙여넣어 실행한다.
3. 콘솔 로그를 확인한다.
   - `[Bot] 시작! ...`가 보이면 정상 시작
   - 진행 중에는 남은 사과 수와 경과 시간이 출력됨

### 중지 방법
- 콘솔에서 아래를 실행:

```js
__stopBot()
```

실행하면 내부 플래그 `window.__botRunning`이 `false`로 바뀌고 루프가 종료됩니다.

### 재실행 방법
- 가장 간단한 방법: `best.js`를 다시 콘솔에 붙여넣어 실행

## 2) 알고리즘 설명

이 코드는 "현재 보드에서 합이 10이 되는 직사각형 선택"을 반복해 점수를 최대화하려고 합니다.

### 전체 흐름
1. 캔버스 찾기
2. React Fiber 트리에서 보드 문자열(170칸 = 17x10) 추출
3. 가능한 직사각형 후보 탐색
4. MCTS 스타일 탐색으로 첫 수 선택
5. 선택한 경로를 드래그 이벤트로 실행
6. 보드 갱신 후 반복

---

### A. 보드 인식
- `getCanvas()`
  - 문서의 `canvas` 중 크기가 충분한 요소를 보드 후보로 선택
- `getBoard()`
  - 캔버스의 React Fiber 내부 상태를 따라가며 보드 데이터 탐색
  - 길이 170의 숫자 문자열(`0-9`)을 찾으면 현재 보드로 사용
- `boardToGrid()`
  - 문자열을 `Uint8Array(170)`로 변환해 연산 비용을 줄임

### B. 후보 직사각형 생성 (findAllRects)
이 단계는 한마디로 "지금 당장 지울 수 있는 사각형을 전부 찾는 과정"입니다.

쉽게 말하면:
1. 보드에서 만들 수 있는 모든 사각형을 하나씩 다 본다.
2. 그 사각형 안의 숫자를 더한다. 단, 0은 빈칸이라서 합산에서 제외한다.
3. 합이 정확히 10이면 "지울 수 있는 후보"로 등록한다.

왜 모든 사각형을 보나?
- 사람은 눈대중으로 몇 개만 보지만, 봇은 놓치지 않기 위해 가능한 사각형을 전부 확인합니다.

작은 예시:
- 어떤 사각형 안에 숫자 4, 3, 3 이 있고 나머지는 0이면 합은 10입니다.
- 이 사각형은 유효 후보가 됩니다.
- 반대로 5, 4, 2 라면 합이 11이라 후보가 아닙니다.

이 코드의 중요한 포인트:
- 사각형 안에서 실제로 지워지는 칸만 따로 저장합니다. (cells)
- 나중에 후보끼리 겹치는지 확인할 때도 사각형 전체가 아니라 cells 기준으로 겹침을 판단합니다.
- 그래서 빈칸(0)이 겹친다는 이유로 좋은 후보를 버리는 일이 줄어듭니다.

성능 최적화:
- 사각형이 전부 숫자로 꽉 찬 상태에서 합이 이미 10을 넘으면, 더 크게 확장해도 절대 10이 될 수 없습니다.
- 이 경우는 바로 탐색을 중단해 시간을 아낍니다.

### C. 1차 선택: MCTS 스타일 탐색으로 첫 수 고르기 (mctsSearch)
이 단계는 "후보가 너무 많을 때, 첫 수를 가장 유리하게 고르는 과정"입니다.

왜 첫 수가 중요하나?
- 같은 보드라도 첫 수를 어떻게 두느냐에 따라 이후에 만들 수 있는 합 10 조합이 크게 달라집니다.
- 즉, 첫 수 하나가 마지막 총 제거 개수에 큰 영향을 줍니다.

동작 순서:
1. 먼저 후보를 정렬합니다.
   - 많이 지우는 후보 우선
   - 같은 개수면 밀도 높은 후보 우선
   - 그래도 같으면 작은 사각형 우선
2. 상위 후보 일부(최대 40개)만 대상으로 삼습니다.
3. 각 후보를 "첫 수"라고 가정하고 가상 플레이를 여러 번 돌립니다.
4. 가상 플레이 결과, 평균적으로 가장 많이 지우는 후보를 실제 첫 수로 선택합니다.

가상 플레이에서 보는 값:
- 첫 수로 지운 개수
- 그 다음 연쇄적으로 더 지울 수 있는 개수
- 둘을 합친 총 제거 수를 점수로 사용

UCB 방식(쉽게 설명):
- 이미 성능이 좋아 보이는 후보는 더 자주 확인하고,
- 아직 충분히 시험하지 않은 후보도 가끔 뽑아서 가능성을 확인합니다.
- 즉, "좋아 보이는 것 활용"과 "새로운 가능성 탐색"을 동시에 하는 균형 전략입니다.

정리:
- 완전한 트리 MCTS 구현은 아니고,
- 상위 후보를 대상으로 반복 시뮬레이션을 돌려 평균 성능이 좋은 첫 수를 찾는 경량 MCTS 스타일입니다.
- 목적은 한 번 많이 지우는 수보다, 최종적으로 더 많이 지우는 첫 수를 고르는 것입니다.

### D. 이후 수: 그리디 (`greedyOnce`)
- 첫 수를 고른 뒤 남은 후보에서 겹치지 않는 직사각형을 순서대로 채택
- 겹침 판정은 직사각형 영역 전체가 아니라 실제 제거 셀(`cells`) 기준
- 한 번 뽑고 끝내지 않고, 보드를 갱신한 뒤 후보를 다시 계산하는 다회 전개로 총 제거 수를 늘림

### E. 입력 실행 (`drag`)
- 셀 좌표를 캔버스 픽셀 좌표로 환산
- `pointerdown -> pointermove(보간) -> pointerup`
- 동일 동작의 마우스 이벤트도 함께 발생시켜 호환성 보강

### F. 시간/루프 제어 (`run`)
- 전체 제한 시간: 약 118초
- 남은 시간과 예상 드래그 시간을 기반으로 탐색 시간(`searchMs`)을 동적 배분
- 보드가 더 이상 변하지 않거나 진행 불가가 반복되면 종료

## 3) 주요 상수

- `COLS = 17`, `ROWS = 10`: 보드 크기
- `TARGET = 10`: 직사각형 유효 합
- `GL, GR, GT, GB`: 캔버스 내부 실제 그리드 경계 비율(좌/우/상/하)

사이트 UI나 캔버스 레이아웃이 바뀌면 이 경계값 보정이 필요할 수 있습니다.

## 4) 주의사항

- 게임 페이지 구조(React 내부 상태, 캔버스 배치)가 바뀌면 동작이 깨질 수 있음
- 브라우저/사이트 이벤트 정책에 따라 포인터 이벤트가 일부 차단될 수 있음
- 자동화 코드는 서비스 정책과 충돌할 수 있으니 사용 환경의 규정을 확인 후 사용 권장
